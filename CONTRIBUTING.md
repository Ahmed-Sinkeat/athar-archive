# المساهمة في أهل الأثر

> هذا الملف هو **الدليل العملي** لإضافة محتوى إلى الأرشيف عبر Git + Pull Request.
> يكمّل [دليل التأليف والنشر](https://github.com/Ahmed-Sinkeat/athar-archive) (التصميم) و**عقد Zod** في
> [`src/content.config.ts`](src/content.config.ts). عند أي تعارض، **العقد في الكود هو الحَكَم** — هذا الملف يطابقه.

**القاعدة الذهبية:** المحتوى أصلُه ملفُّ Markdown في `src/content/`. لا تُحرّر الموقع المنشور ولا قاعدة بيانات.
إن لم يكن التغيير في Markdown، فهو ليس جزءاً من الأرشيف. الموقع كله يُعاد بناؤه من Git وحده.

---

## قبل أن تبدأ

```bash
pnpm install            # تثبيت الاعتماديات
pnpm dev                # خادم التطوير المحلي
pnpm new <entity> <slug> [العنوان]   # توليد ملف جديد بقالب صحيح (السقالة)
pnpm validate:content   # قواعد الربط عبر الكيانات (مراجع، مراسٍ، تفريغ، slug)
pnpm build              # Zod + بناء Astro + فهرسة البحث
pnpm test               # اختبارات vitest
```

طبقتا التحقّق اللتان عليك إرضاؤهما:

| الأمر | يتحقّق من |
|---|---|
| `pnpm build` (يشغّل Zod) | **الحقول الإلزامية لكل كيان**، الأنواع، القوائم المحصورة (enums)، صيغ التاريخ والروابط |
| `pnpm validate:content` | **الربط عبر الكيانات**: صحة المراجع، حلّ المراسي، بوابة التفريغ، صيغة المعرّف |

كلاهما يعمل آلياً على كل Pull Request عبر CI. لا يُدمج شيء وهو أحمر.

## 00 · سير العمل من النص إلى الصفحة

كل محتوى يمرّ بنفس المسار، سواء كتبه عضو فريق أو وصل من متطوّع. **لا استثناءات.**

1. **تهيئة الملف** — `pnpm new <entity> <slug>` أو أنشئ `.md` يدوياً في مجلد الكيان الصحيح، بالـ frontmatter المطلوب (§02).
2. **كتابة المحتوى** — النص بالعربية، مع علامات الأبواب إن طال (§04) ومراسي الشروح إن وُجدت (§05).
3. **التحقّق محلياً** — `pnpm validate:content && pnpm build`؛ يُرفض أي حقل ناقص أو مرجع معلّق أو مرساة لا تطابق موضعاً.
4. **المراجعة (Pull Request)** — تفتح PR؛ يراجعها عضو فريق آخر: الصحة العلمية، الاتساق التحريري، المطابقة للقواعد.
5. **ضبط الحالة والدمج** — عضو الفريق ينقل `status` إلى `published` ويدمج في `main`؛ هذه لحظة النشر.
6. **النشر الآلي** — الدمج يُطلق البناء والنشر على الحافة، ويعيد بناء فهرس البحث.

## 01 · مسار المتطوّع

المتطوّع **لا يلتزم في Git مباشرة**. يرسل نصاً أو ملف Markdown، ويتولّى عضو الفريق تحويله إلى صيغة الأرشيف وفتح/تنسيق الـ PR.

1. **الاستلام** — يصل نص المتطوّع. يُعامَل **مدخلاً غير موثوق** حتى يُراجَع ويُنقّى.
2. **التهيئة** — يحوّله عضو الفريق إلى `.md` صحيح: frontmatter، ضبط المعرّف، ربط الموضوعات، تنسيق الأبيات/الفقرات.
3. **التنقية** — يُزال أي HTML خام أو روابط مشبوهة؛ النص العلمي فقط يبقى. (التنقية الآلية في البناء طبقةٌ ثانية لا بديل عن المراجعة.)
4. **الإسناد** — يُنسب الشرح/الفائدة لصاحبه عبر `person` إن كان منسوباً، أو يُترك إن كان تحريراً تجميعياً (وكان اختيارياً).
5. **المراجعة العلمية** — يتحقّق عضو الفريق من صحة العزو والتخريج قبل ضبط الحالة.

> ⚠️ **تنبيه أمني:** ملف المتطوّع مدخلٌ غير موثوق بالتعريف. لا تُلصِق محتواه كما هو دون قراءة كاملة وتنقية. هذا أهم بندٍ
> لا يحميك منه المعمار الثابت تلقائياً. راجع [سياسة المحتوى المسموح](#ما-الذي-يُسمح-في-المحتوى) و[`docs/governance.md`](docs/governance.md).

## 02 · الـ frontmatter لكل كيان

كل ملف يبدأ بكتلة YAML بين `---`. **الحقول المشتركة** لكل الكيانات:

| الحقل | النوع | إلزامي | ملاحظة |
|---|---|:---:|---|
| `title` | نص | ✅ | عنوان الكيان (لا يكون فارغاً) |
| `status` | `draft` \| `review` \| `published` \| `archived` | ✅ | §06 |
| `published_at` | تاريخ `YYYY-MM-DD` | ✅ | يُضبط مرة عند أول نشر |
| `updated_at` | تاريخ | — | اختياري |
| `aliases` | قائمة معرّفات | — | معرّفات قديمة → إعادة توجيه 301 |

> 🛑 **تصحيح مهم لدليل الإصدار 1.0:** لا يوجد حقل `id` ولا `slug` ولا `author` في الـ frontmatter.
> **المعرّف هو اسم الملف** (§03)، وحقل المؤلف اسمه **`person`** لا `author`. واتبع دائماً ما يقبله العقد أدناه.

أمثلة لكل كيان (يولّدها `pnpm new <entity> <slug>` جاهزةً):

### الكيانات الأساسية

```yaml
# person — الشخص   (src/content/person/<slug>.md)
title: ابن تيمية
status: draft
published_at: 2026-06-21
# bio | born | died | location | image  — كلها اختيارية
```

```yaml
# subject — التصنيف   (الجذر؛ الموضوعات تنتمي إليه)
title: العقيدة
status: draft
published_at: 2026-06-21
# description — اختياري
```

```yaml
# topic — الموضوع
title: الأسماء والصفات
subject: aqeedah          # ← إلزامي: معرّف تصنيف موجود
status: draft
published_at: 2026-06-21
# description — اختياري
```

```yaml
# book — الكتاب
title: العقيدة الواسطية
person: ibn-taymiyyah     # ← إلزامي: معرّف المؤلف/الشارح
topics: [al-asma-was-sifat]   # اختياري، 1–5
status: draft
published_at: 2026-06-21
# sharh_of: al-wasitiyyah       # اختياري: إذا كان الكتاب شرحاً لكتاب آخر
# transcript_status: قيد المراجعة # اختياري: للشروح المفرغة (دروس مفرغة)
# hadith_category: امهات الكتب    # اختياري: لكتب الحديث (امهات الكتب | أجزاء حديثية | تخريج | علل | عام)
# description | edition | cover | attachments — اختيارية
```

```yaml
# poem — المنظومة   (verse_count و opening_verse مُشتقّان من المتن، لا تكتبهما)
title: ألفية ابن مالك
person: ibn-malik-al-nahwi   # ← إلزامي
topics: [al-nahw-al-muyassar]
status: draft
published_at: 2026-06-21
# description | attachments — اختيارية
```



```yaml
# question — المسائل   (موضوعات فقط، لا تصنيف ولا وسوم)
title: حكم كذا
topics: [al-asma-was-sifat]   # ← إلزامي، 1–5 (التصنيف يُشتق منها)
status: draft
published_at: 2026-06-21
# person — اختياري: المُجيب
```

```yaml
# benefit — الفائدة
title: فائدة في كذا
person: ibn-taymiyyah     # ← إلزامي: صاحب الفائدة
topics: [al-asma-was-sifat]
status: draft
published_at: 2026-06-21
# source_type: book        # اختياري: book | article | poem
# source_id: <source-id>   #         (لازمٌ معه source_type)
```

```yaml
# article — المقالة
title: مقالة في كذا
person: ibn-taymiyyah     # ← إلزامي: الكاتب
topics: [al-asma-was-sifat]
status: draft
published_at: 2026-06-21
# audio | description | attachments — اختيارية
```

### الكيانات المضمَّنة (لا صفحة مستقلة لها)

```yaml
# audio — الصوتية   (تُعرض داخل الكتاب/المتن/المقالة)
title: تلاوة البيقونية
source_type: poem         # ← إلزامي: book | poem | article
source_id: al-bayquniyyah # ← إلزامي: معرّف المصدر
url: https://r2.ahlalathar.com/audio/al-bayquniyyah.opus   # ← إلزامي
format: opus              # opus | mp3 (الافتراضي opus)
status: draft
published_at: 2026-06-21
# duration | size_bytes — اختيارية
# يمكن إضافة أكثر من صوتية لنفس المتن/المنظومة (تسجيلات أو شُرّاح متعددون)؛
# يظهر حينها للقارئ سهمٌ صغيرٌ (قائمة) لاختيار التسجيل — يقتصر هذا على المتون والمنظومات.
```

```yaml
# annotation — الشرح/الحاشية/الإعراب/التخريج   (مضمَّن عند مرساته)
title: شرح البيت الأول
target_type: poem         # ← إلزامي: book | poem
target_id: al-bayquniyyah # ← إلزامي: معرّف المتن/المنظومة
anchor: v1                # ← إلزامي: v{n} لبيت، أو p{n}/{#id} لفقرة كتاب (§05)
kind: شرح                 # شرح | حاشية | إعراب | تخريج (القيمة عربية)
status: draft
published_at: 2026-06-21
# annotator: <person-id>   # اختياري
```

```yaml
# announcement — الإعلان   (زينة الصفحة الرئيسة فقط)
title: إعلان
status: published
published_at: 2026-06-21
priority: 5               # 1–10 (الافتراضي 5)
# body | expires_at — اختياريان
```

```yaml
# highlight — مختار الأسبوع   (آية/حديث/بيت — زينة الصفحة الرئيسة، تدور أسبوعياً)
title: حديث النية          # عنوانٌ مختصر للإدارة
kind: حديث                # ← إلزامي: آية | حديث | بيت (القيمة عربية)
reference: متفق عليه — رواه البخاري ومسلم   # المصدر/التخريج (اختياري): «البقرة ٢٥٥»، اسم الناظم…
status: published
published_at: 2026-06-22
# النصّ نفسه (الآية/الحديث/البيت) يُكتب في body بعد الـ frontmatter
```

> الصفحة الرئيسة تعرض مختاراً واحداً من كلِّ نوع، وتدور أسبوعياً بين المنشور.
> أمّا متون الحديث كاملةً (نحو **الأربعون النووية**) فتُضاف ككتاب `kind: متن`، وتُخرَّج
> أحاديثُه بشروحٍ من نوع `تخريج` (انظر §05).

> **قاعدة المصدر:** كل `(source_type, source_id)` و`target_id` يجب أن يحلّ إلى كيان **موجود**.
> مرجعٌ معلّق = **فشل بناء**. وكيانٌ منشور لا يجوز أن يشير إلى آخر في حالة `draft`.

## 03 · قواعد المعرّف والـ slug

المعرّف عقدٌ دائم. خطأ فيه يُكلِّف لاحقاً.

- **المعرّف = اسم الملف** (بلا `.md`). لا يوجد حقل `id`/`slug` في الـ frontmatter.
- لاتيني، صغير الأحرف، بشرطات (kebab-case)، وصفي ومستقر: `al-wasitiyyah` لا `book1`.
  النمط المفروض: `^[a-z0-9]+(--?[a-z0-9]+)*$`.
- **ثابت للأبد بعد النشر.** إن اضطُررت لتغيير معرّف منشور، انقل القديم إلى `aliases` لتوليد إعادة توجيه — لا تكسر رابطاً داخلاً أبداً.
- **التوابع تُبنى من الأب** بفاصل `--`: `<target_id>--<anchor>--<kind>` للشرح، مثل `alfiyyah-ibn-malik--v1--sharh`.
- **اسم ملف الشرح:** `<target_id>--<anchor>--<kind>` بـ kind لاتيني (`sharh` \| `hashiya` \| `iraab` \| `takhrij`)،
  مثل `alfiyyah-ibn-malik--v1--sharh.md`. لاحظ أن حقل `kind` **داخل** الملف يبقى عربياً (`شرح`)؛ الاسم لاتيني لأن المعرّف لاتيني.

## 04 · علامات الأبواب والتقطيع

الأبواب **مشتقّة من علامات داخل ملف واحد**، لا من ملفات منفصلة ولا تقطيع يدوي. أنت تكتب البنية، والبناء يقطّع.

- **علامة الباب** = عنوان Markdown من مستويين بالضبط: `## عنوان الباب`.
- **بيت المنظومة** = سطر واحد، يُفصل شطراه بـ ` --- ` (أو ` ... ` أو `…`). الترقيم تسلسلي آلي عبر المتن كله.
- **فقرة الكتاب** = كتلة نصّ مفصولة بسطر فارغ. تأخذ مرساة آلية `p1`, `p2`… أو مرساة صريحة في آخرها `{#my-id}`.
- **لا تقطّع يدوياً:** ملفٌ واحد يبقى مصدر الحقيقة؛ التقطيع نتيجةٌ آلية للعتبة وقت البناء.

| الحالة | السلوك الآلي |
|---|---|
| منظومة دون 200 بيت | صفحة واحدة، الأبواب فواصل عنوانية داخلها |
| منظومة فوق 200 بيت | كل باب صفحة `/poem/:slug/:chapter` + فهرس أبواب |
| كتاب فوق العتبة (~6000 كلمة أو 8 أبواب) | كل باب صفحة `/book/:slug/:chapter` + فهرس أبواب |

العتبات قابلة للضبط في [`ahlalathar.config.ts`](ahlalathar.config.ts).

## 05 · كتابة الشروح (Annotation)

الشرح/الحاشية/الإعراب/التخريج كيانٌ مستقل مرتبط بموضع، يُكتب في ملفه الخاص بمرساة تشير إلى موضع **موجود فعلاً** في المصدر.

- المرساة `anchor` يجب أن تطابق موضعاً حقيقياً:
  - منظومة → رقم بيت ضمن النطاق: `v5`.
  - كتاب → معرّف فقرة موجود: `p12` أو `{#id}` صريح.
  - **مرساة معلّقة = فشل بناء.**
- `target_type` إمّا `book` أو `poem` فقط.
- `kind` قيمته **عربية**: `شرح` \| `حاشية` \| `إعراب` \| `تخريج` (الافتراضي `شرح`). العرض يتكفّل بالتسمية واللون.
- الشرح يُعرض مضمَّناً عند مرساته ويُشارَك عبر `/poem/<id>#v5-...` — لا صفحة مستقلة له.

## 06 · الحالة وبوابات النشر

الحالة تحكم دورة حياة الكيان، والبوابات تمنع نشر محتوى ناقص. كلاهما مرمَّز في Zod/المُتحقّق لا متروكٌ للذاكرة.

| الحالة | المعنى |
|---|---|
| `draft` | قيد الكتابة؛ لا يُعرض ولا يُفهرس. لا يُشار إليه من كيان منشور |
| `review` | جاهز للمراجعة؛ بانتظار إقرار عضو فريق |
| `published` | منشور وحيّ؛ ينقله عضو فريق فقط، وعند الدمج يصبح صفحة |
| `archived` | مؤرشف؛ يبقى رابطه حيّاً (لا يُكسر) لكن لا يظهر في القوائم النشطة |

**بوابات مرمّزة (قواعد بناء لا أعراف):**
1. الأقسام الفارغة ممنوعة.
2. لا يُنشر كيانٌ يشير إلى تابع في حالة `draft`.
3. نقل أي شيء إلى `published` وصلاحية الدمج في `main` **محصوران بأعضاء الفريق** (عبر CODEOWNERS + حماية الفرع — [`docs/governance.md`](docs/governance.md)).

## 07 · قائمة المراجعة قبل النشر

يمرّ عليها المراجع قبل ضبط الحالة إلى `published` والدمج (وهي نفسها قالب الـ Pull Request):

- [ ] الـ frontmatter كامل وكل الحقول الإلزامية موجودة.
- [ ] المعرّف (اسم الملف) لاتيني وصفي ثابت ومطابق للنمط.
- [ ] كل `(source_type, source_id)` و`target_id` يحلّ إلى كيان منشور.
- [ ] كل مرساة Annotation تطابق موضعاً موجوداً فعلاً في المصدر.
- [ ] الموضوعات 1–5، حقيقية لا وسوم.
- [ ] الفائدة مقتطفٌ مركّز لا مقالة، والمقالة مستقلة لا سلسلة.
- [ ] محتوى المتطوّع قُرئ كاملاً ونُقّي من HTML خام وروابط مشبوهة.
- [ ] الصحة العلمية: العزو والتخريج مدقَّقان.
- [ ] للدرس: التفريغ موجود.
- [ ] `pnpm validate:content && pnpm build` يمرّان أخضر محلياً.

## ما الذي يُسمح في المحتوى

اكتب **Markdown** (GitHub-flavored). تُحوَّل كتابتك إلى HTML آمن عبر `rehype-sanitize`
(انظر [`src/lib/sanitize-schema.ts`](src/lib/sanitize-schema.ts)):

- **يُسمح:** المجموعة الآمنة المعيارية (عناوين، فقرات، قوائم، جداول، روابط، تشديد، اقتباس، شيفرة)، وعلى أي عنصر السماتُ
  `id` و`class` و`dir` و`lang` (لازمةٌ لمراسي الفقرات والنصوص ثنائية الاتجاه).
- **يُجرَّد دائماً:** `<script>` و`<style>` ومعالجات الأحداث (`onclick`…) وأي وسمٍ خارج المجموعة الآمنة. لا تعتمد عليها.
- **القاعدة العملية:** اكتب بالعربية بصيغة Markdown؛ تجنّب HTML الخام. إن لزمتك جهة نص مختلفة استعمل `<span dir="ltr">…</span>`.

> هذه السياسة تُغلق المسألة #10 في [`docs/issue.md`](docs/issue.md): المجموعة المسموحة موثّقة، وأي توسيع لها يُراجَع هنا أولاً.
