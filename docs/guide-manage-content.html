Adding, Editing & Deleting Content

Content (books, poems, people, articles, questions, etc.) is managed from a single CMS at /admin (Sveltia) — full create/edit/delete with real form fields, no copy-paste step.

1. Open the admin panel

Go to yoursite.com/admin and sign in with GitHub (any account with write access to the repo).

Every save is a real commit straight to the live main branch — there is no draft/review step, changes go live on the next automatic rebuild (a few minutes).

2. What you can manage

CollectionWhat it is
BookFull books/texts (matn, etc.)
PoemPoetic works (منظومات)
ArticleWritten articles
QuestionQ&A / fatwa-style entries
PersonScholar/author profiles
Subject / Topic / TermClassification & glossary entries
AudioAudio recordings — drop a file directly or paste an R2/external URL
AnnotationNotes attached to a piece of content

Not editable here: the Quran collection (114 surahs, auto-generated) and «الفوائد / كُناشة» (device-local reader bookmarks, not editorial content — see docs/technology-stack.md).

3. Add, edit, or delete an entry

Add: open the collection in the sidebar → "Create" → fill in the fields → Save.

Edit: open the collection → click the existing entry → change fields → Save.

Delete: open the entry → the Delete option is near the Save button.

Fields every entry shares

FieldNotes
slugBecomes part of the URL. Don't change it after the page is published — old links will break.
statusdraft / review / published / archived — only published shows on the live site.
published_at / updated_atDates shown/sorted on the site.
aliasesAlternate URLs that redirect to this entry.
Gotchas — fields that look free-text but must match an exact value (get these wrong and the entry silently stops showing up in filters/search):
  • Topic → subject is required.
  • Question → topics needs at least one.
  • Annotation needs target_type, target_id, and anchor together.
  • Audio needs source_type, source_id, and url together.
  • Fields like era, ṭabaqa, kind, genre, work_type, grade, rutba are typed as plain text but the site expects specific Arabic values — copy the value from a similar existing entry rather than typing a new one from scratch.

Undo a mistake

Every save is a GitHub commit. If something goes wrong, the repo owner can revert the commit in GitHub's history — no content is permanently lost.
