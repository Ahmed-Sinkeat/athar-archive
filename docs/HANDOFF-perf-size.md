# HANDOFF — perf & size fixes (approved 2026-07-02)

> For the implementing agent. Plan reviewed and approved by Ahmed on 2026-07-02.
> Scope: this repo (athar-archive) only — no engine changes, no stack changes.

## Context

Two symptoms: **book pages open slowly** and **the deploy keeps growing** (`dist/client`
is 162MB, was ~90MB). A full review of the stack (Astro 6 + Cloudflare Workers Static
Assets, Markdown collections, no DB, vanilla TS, Pagefind) concluded the stack is right;
both symptoms come from two data-flow decisions:

1. **Surah pages inline the full tafsir of every verse.** `src/pages/quran/[surah].astro`
   imports the 93MB `src/data/quran-tafsir-index.json` (6,236 verse keys, 16,096 notes)
   and renders every note as hidden `ann-pack` divs. Result: `dist/client/quran` = 114MB,
   surah 2 = **7.8MB of HTML**, and it grows linearly with each new tafsir source.
2. **Book chapter pages re-read and re-split the whole book per cold request.**
   `src/pages/book/[slug]/[chapter].astro` → `loadContentBody()` fetches the *entire*
   book from ASSETS, `analyzeBook()` re-chunks it, then ~6 regex passes over the full
   HTML, plus `loadGraph()` per request. `tafsir-ibn-kathir.md` is 25,967,728 bytes —
   **99% of the 26,214,400-byte (25 MiB) per-asset limit**; the next growth breaks
   `wrangler deploy`. Edge cache (`src/middleware.ts`) is per-colo, 1-day TTL, so cold
   renders are frequent.

Explicitly **not** doing: Postgres/D1/KV (read-only build-time content needs no runtime
store), prerendering all ~44k chapter pages (blows the 20,000-file deploy limit),
tafsir range-dedup (no quota cares about total size, only per-file and count).

## M1 — Per-verse tafsir fragments

- New build step (extend `scripts/copy-content-assets.mjs` or a sibling script run after
  `astro build`): render `quran-tafsir-index.json` → static fragments
  `dist/client/tafsir-frag/<surah>/<ayah>.html`, sanitized at build via the existing
  `markdownToSafeHtml` (`src/lib/sanitize.ts`). ~6,236 files — count is per-verse, so it
  does **not** grow when new tafsir sources land.
- `src/pages/quran/[surah].astro`: drop the inline `ann-pack` divs and the 93MB import;
  keep the verse-number button, add `data-ann-src="/tafsir-frag/<surah>/<ayah>.html"`.
- `src/scripts/reader.ts` `openSheet()`: if the pack isn't in the DOM but the trigger has
  `data-ann-src`, fetch the fragment once, inject as a pack, open. (~15 lines; the
  synthetic-pack pattern at reader.ts:734 shows how.)
- Expected: surah 2 page 7.8MB → ~150KB; `dist/client` −110MB. SEO: tafsir text stays
  crawlable on the tafsir books' own reading pages.

## M2 — Per-chapter book assets

- Extend `copy-content-assets.mjs`: for each published book, run `analyzeBook`
  (`src/lib/chunk.ts`) **once at build** and write
  `dist/client/content/book/<slug>/<chapter>.md` + `<slug>.chapters.json` (title/slug
  manifest for the TOC). For chunked books, stop copying the whole-book `.md`
  (removes the 26MB asset and the 25 MiB-limit risk).
- `src/pages/book/[slug]/[chapter].astro`: read the one chapter file + manifest instead
  of whole-book-then-split. `chunk.ts` stays the single source of truth for the split —
  same code, moved from request time to build time.
- Leave `loadGraph()` per-request as is (its `ponytail:` comment already names the
  upgrade path); after this change it's the remaining cost and likely fine.
